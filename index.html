Growver

Autonomous tree-planting rover · 14-engineer team
Jan 2026 – May 2026

A battery-powered rover that drives itself across terrain and runs a fully autonomous planting sequence — digging, placing a seedling, dispensing soil, and compacting it. On a 14-engineer team spanning mechanical, electrical, and software, I was the sole firmware engineer and owned the entire software stack end to end.

Architecture

The firmware runs on an ESP32 under Zephyr RTOS. I built it around a single context object — one mcu_t pointer created at startup and threaded through the rest of the program, carrying the current state and the objects for every motor. Centralizing state this way kept ownership clear and made the system easy to reason about as it grew.

Motor abstraction

The rover has 7 unique motors, but the rest of the code never touches their wiring. Each motor is a struct that records its gearbox ratio, current internal state, the pins it lives on, and the physical driver it uses. All of that hides behind just two API calls — set_motor_speed() and set_step_motor() — so any motor can be commanded with one line.

State machine

A 6-state finite state machine drives the mission, moving through the phases in order:

  • Bootup — bring the system up and initialize hardware
  • Idle — wait, ready for an operator command
  • Drive — movement across the terrain
  • Dig — break ground for the seedling
  • Place — set the seedling into the hole
  • Tamp — dispense soil and compact it

Concurrency

I split the work across 4 Zephyr threads so nothing blocks anything else: a main thread running the rover’s operations, one receiving telemetry, one reading the battery management system over I²C, and one driving the LED user interface. The BMS talks to a student-built 24-cell lithium-ion pack, and the LED and buzzer feedback report the rover’s current state to the operator at a glance.

Senior Capstone · Firmware Engineer

RF Communication Device

Wireless embedded device · Engineering Design phase
Jul 2026 – Present

My senior capstone: an RF communication device currently in the engineering design phase. The plan centers on an microcontroller programmed in C, with peripheral devices coordinated over an I²C bus.

Where it stands

The project is in active engineering design — defining the architecture, selecting components, and laying out how the firmware will drive the radio link and the sensors and peripherals around it. As the firmware engineer I’m focused on the embedded side: a ZephyrOS C application on the mcirocontroller with I²C as the backbone for talking to attached hardware.

Planned direction

  • Microcontroller in C as the core controller for the device.
  • I²C to coordinate sensors and peripherals on the board.
  • An RF link as the headline capability — the design work this fall will turn that goal into a working prototype.
